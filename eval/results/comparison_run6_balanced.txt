   the methodology prompt more faithfully on formulaic tasks.

3. Hard difficulty improved most across versions (V1: 2/8, V4: 5/8).
   Prompt methodology + context management compound on vague symptoms.

4. Evidence quality is the most stable metric across runs.
   RCA and hit rate have n=1 judge variance.

5. Multi-turn latency is noisy (rate limit dominated). Single-turn
   latency (224s→147s, -34%) is the reliable speed signal.

6. Token reduction (40.7K→28.3K, -31%) comes from three layers:
   V2 prompt (converge faster), V3 compaction (clear stale data),
   V4 routing (cheaper model = leaner sweep output).

════════════════════════════════════════════════════════════════
  WHY gpt-4.1 OUTPERFORMS gpt-5.4 ON THE FIRST SWEEP
════════════════════════════════════════════════════════════════

Root cause: an interaction between model behavior and incomplete
signal discovery in the topology injection.

WHAT HAPPENS
  Before the first LLM call, agent.py injects a service topology
  via list_services. This tool queries label/tag discovery APIs from
  Mimir, Tempo, and Loki to report which signals each service has.
  However, only Mimir's API returns data. Loki and Tempo's discovery
  APIs return empty responses in the grafana/otel-lgtm all-in-one
  image — the labels API returns {"status":"success"} with no data
  payload, despite data being fully queryable via LogQL/TraceQL.

  Result: topology shows every service as "metrics" only.
  The data exists in all three backends, but discovery can't see it.

HOW THE MODELS DIVERGE
  gpt-5.4 reads the topology, reasons about it, and decides:
    "topology shows only metrics-enabled services, so querying
    logs and traces would be wasteful — I'll focus on metrics."
  18/32 V3 scenarios explicitly state this reasoning.
  It mentions all 3 signals in 22/32 openings but actually
  checks all 3 in only 11/32 — low follow-through.

  gpt-4.1 reads the same topology but follows the prompt's
  Step 2 literally: "query metrics, logs, and traces simultaneously."
  It doesn't try to optimize based on topology.
  It checks all 3 signals in 26/32 scenarios — high follow-through.

SIGNAL DISCOVERY GAP
  list_services relies on label/tag discovery APIs to determine
  signal availability. In the grafana/otel-lgtm:0.8.1 image,
  Loki and Tempo's discovery APIs return empty despite service_name
  being correctly configured as an indexed label. Only Mimir's
  Prometheus-compatible API works reliably. The tool silently
  treats missing data as "no services" — no warning logged.

  Fix options (not implemented — documented as future work):
  a) Infra: investigate otel-lgtm image labels API behavior
  b) Code: probe backends with limit=1 test queries instead of
     relying on discovery APIs

GENERAL PRINCIPLE
  In agentic systems, match model capability to subtask type:
  - Execution tasks (structured, formulaic): use a model with
    high instruction-following fidelity (gpt-4.1). The prompt
    already encodes the right strategy — don't second-guess it.
  - Reasoning tasks (correlation, diagnosis): use a model with
    deep reasoning capability (gpt-5.4). Autonomous planning
    is an asset when the task requires judgment.
  gpt-5.4's "intelligence" becomes a liability on the sweep
  because it second-guesses correct methodology based on
  incomplete context. gpt-4.1 treats the prompt as a contract.
